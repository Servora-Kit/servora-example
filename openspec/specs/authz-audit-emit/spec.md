# Spec: authz-audit-emit

## Purpose

Defines requirements for the `authz-audit-emit` capability.

## Requirements

### Requirement: Authz middleware supports audit recorder injection

`pkg/authz` SHALL provide a `WithDecisionLogger(fn func(ctx context.Context, detail DecisionDetail))` option that accepts a callback function for authorization decision logging. This replaces the previous `WithAuditRecorder(r *audit.Recorder)` option.

The middleware SHALL NOT directly depend on `pkg/audit`. Services bridge the audit system via a closure:

```go
authz.WithDecisionLogger(func(ctx context.Context, d authz.DecisionDetail) {
    recorder.RecordAuthzDecision(ctx, d.Operation, actorFromCtx, audit.AuthzDetail{...})
})
```

When no `DecisionLogger` is configured, the middleware SHALL function normally without any logging.

#### Scenario: Decision logger via closure
- **WHEN** `authz.Server(authorizer, authz.WithDecisionLogger(fn))` is called
- **THEN** the middleware SHALL invoke `fn` after each authorization check

#### Scenario: Nil decision logger is safe
- **WHEN** `authz.Server(authorizer)` is called without `WithDecisionLogger`
- **THEN** the middleware SHALL function normally without invoking any logging callback

### Requirement: protoc-gen-servora-authz generates func returning copy

`cmd/protoc-gen-servora-authz` SHALL generate `authz_rules.gen.go` files with:
- An unexported `var _authzRules` holding the rule map
- An exported `func AuthzRules() map[string]authz.AuthzRule` that returns a shallow copy
- The file header `// Code generated by protoc-gen-servora-authz. DO NOT EDIT.`

This is a **BREAKING** change from the current `var AuthzRules` output.

#### Scenario: Generated func returns independent copy

- **WHEN** a caller invokes `AuthzRules()` twice and modifies one copy
- **THEN** the other copy and the internal `_authzRules` SHALL remain unchanged

#### Scenario: Existing IAM rules regenerated as func

- **WHEN** `make api` regenerates `api/gen/go/servora/iam/service/v1/authz_rules.gen.go`
- **THEN** the file SHALL contain `func AuthzRules()` instead of `var AuthzRules`

### Requirement: WithAuthzRules accepts func signature

`pkg/authz` SHALL provide `WithAuthzRulesFunc(fn func() map[string]AuthzRule)` option that accepts the codegen-generated function. The existing `WithAuthzRules(map[string]AuthzRule)` SHALL be retained for backward compatibility with hand-written rules.

#### Scenario: Middleware configured with func rules

- **WHEN** `Authz(WithFGAClient(c), WithAuthzRulesFunc(iampb.AuthzRules))` is called
- **THEN** the middleware SHALL call the function once to obtain the rules map and use it for authorization checks

#### Scenario: Middleware configured with static rules

- **WHEN** `Authz(WithFGAClient(c), WithAuthzRules(staticMap))` is called
- **THEN** the middleware SHALL use the static map directly (backward compatible)

### Requirement: Authz middleware emits authz.decision event after Check

After each authorization Check (whether allowed, denied, or errored), the middleware SHALL call the configured `DecisionLogger` with a `DecisionDetail` containing Operation, Subject, Relation, ObjectType, ObjectID, Allowed, Err, and CacheHit.

#### Scenario: Allowed check triggers logger
- **WHEN** a request passes authorization (authorizer returns `true, nil`)
- **THEN** the middleware SHALL call the `DecisionLogger` with `Allowed: true`

#### Scenario: Denied check triggers logger
- **WHEN** a request fails authorization (authorizer returns `false, nil`)
- **THEN** the middleware SHALL call the `DecisionLogger` with `Allowed: false` before returning the permission error

#### Scenario: Check error triggers logger
- **WHEN** the authorizer returns an error
- **THEN** the middleware SHALL call the `DecisionLogger` with `Allowed: false, Err: <error>`

#### Scenario: No logger skips emission
- **WHEN** the middleware has no `DecisionLogger` configured
- **AND** a request is processed
- **THEN** authorization SHALL proceed normally without any logging

### Requirement: Authz middleware passes full principal to Authorizer
The middleware SHALL construct the full principal string using `string(actor.Type()) + ":" + actor.ID()` before passing it to `authorizer.IsAuthorized()`. It SHALL NOT hardcode any actor type prefix.

#### Scenario: User actor principal construction
- **WHEN** a request from a user actor with Type `"user"` and ID `"abc-123"` is authorized
- **THEN** the middleware SHALL call `IsAuthorized` with subject `"user:abc-123"`

#### Scenario: Service actor principal construction
- **WHEN** a request from a service actor with Type `"service"` and ID `"order-svc"` is authorized
- **THEN** the middleware SHALL call `IsAuthorized` with subject `"service:order-svc"`

### Requirement: Authz middleware does not hardcode allowed actor types
The middleware SHALL NOT hardcode `a.Type() != actor.TypeUser` as a rejection condition. It SHALL reject `anonymous` actors by default and allow all other actor types (including `service`) to proceed to the `Authorizer`.

#### Scenario: Service actor allowed to Check
- **WHEN** a service actor makes a request to a CHECK-mode operation
- **THEN** the middleware SHALL construct the principal and call the `Authorizer`

#### Scenario: Anonymous actor rejected
- **WHEN** an anonymous actor makes a request to a CHECK-mode operation
- **THEN** the middleware SHALL return 403 AUTHZ_DENIED without calling the `Authorizer`
